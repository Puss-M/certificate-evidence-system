# 项目协作规则：可信证书存证系统

## 1. 项目定位

本项目选题为“创新项目 1：基于区块链的实训证书与学业证明存证系统”。实际开发时统一按以下定位推进：

> 证书生命周期管理系统 + 哈希存证验真系统，并预留向“可信学业凭证 / 诚信简历”扩展的抽象。

当前阶段仍以创新项目 1 为主线：证书是最小可信凭证单元。创新项目 3 的思想只吸收为后续扩展方向：将课程证书、竞赛证书、科研经历、任职证明等统一抽象为 `CredentialItem`，未来可通过 Merkle Tree 支持学生选择性披露，生成可信简历。

区块链、哈希链、Merkle Tree、DID/VC、选择性披露等只作为可信机制增强，不作为两周实训的主体目标。所有开发优先保证可运行、可演示、可测试、可说明的证书闭环。

## 2. 两周内必须完成的最小闭环

必须优先完成以下链路，任何提高项都不能影响主链路稳定性：

1. 教师或管理员配置证书模板。
2. 通过 Excel 导入学生名单。
3. 系统批量生成证书编号、证书 PDF 和二维码。
4. 对证书正文或 PDF 计算 SHA-256 哈希。
5. 将证书哈希、机构、时间戳、状态、回执写入本地哈希链或本地测试链。
6. 学生查询、下载、分享证书二维码。
7. 第三方扫码验真，展示哈希校验结果、证书状态和存证回执。
8. 管理员撤销或补发证书，系统保留操作日志。

## 3. 非目标与降级策略

- 不连接真实主网，不产生链上费用。
- 不承诺“绝对防伪”，只证明某个证书文件版本在存证后未被篡改，并结合系统状态判断是否有效。
- 不把 Merkle Tree 解释成万能防伪。哈希和 Merkle 解决的是“存证后是否被改、是否属于已承诺集合”；现实真实性和当前有效性还依赖学校身份、签名权限、审核流程、状态查询、撤销机制和审计日志。
- DID/VC、多机构协同、真实电子签章、真实短信邮件、复杂权限组织架构、诚信简历完整选择性披露都属于提高项或后续扩展。
- 如果真链接入拖慢进度，立即降级为本地哈希链；答辩时解释为“教学版可信存证模拟”。

## 4. 创新项目 3 思想的吸收边界

本项目可以吸收“大学生诚信简历系统”的思想，但只能作为抽象设计和扩展叙事，不能压垮创新项目 1 的主线。

### 可以吸收

- 将证书抽象为一种可信凭证：`Certificate` 是当前实现对象，`CredentialItem` 是未来统一抽象。
- 在数据库或接口设计中预留凭证类型字段，例如课程证书、竞赛证书、培训证明、科研经历、任职证明。
- 在文档中说明：多张证书或多条凭证未来可组成 Merkle Tree，学生可只披露其中部分凭证，并提供 Merkle Proof 给第三方验证。
- 在提高项中加入“简化版可信简历”：学生选择若干已发证书生成一个分享页或二维码。

### 暂不进入两周主线

- 不要求实现完整诚信简历系统。
- 不要求实现复杂 Merkle Proof 管理页面。
- 不要求实现企业端多项凭证批量核验。
- 不要求实现真正匿名或零知识证明。
- 不要求支持成绩、竞赛、科研、任职等多类复杂材料的完整审核工作流。

### 必须讲清楚的可信边界

- Merkle Proof 必须锚定到可信 Root，例如学校签名 Root、链上回执或本地哈希链回执；否则 Proof 本身也可能被伪造。
- 没有学校签名、链上/本地哈希链回执或状态表锚定的 Merkle Root / 凭证分享页，只能算展示页，不能算可信证明。
- 旧 Proof 只能证明“曾被纳入”，不能单独证明“现在仍有效”；当前有效性必须查询状态表、撤销记录或最新承诺 Root。
- 选择性披露只能减少暴露，不等于完全匿名；如果某个经历很独特，仍可能被反推出身份或履历背景。

## 5. 技术路线建议

除非团队已有明确技术栈，否则优先使用：

- 后端：FastAPI，作为本项目后端主线技术栈。
- 数据库：MySQL。
- 前端：Vue 3，优先保证管理员端、学生端和公共验真端演示流畅。
- 文档生成：reportlab、docxtpl 等 Python 生态稳定方案。
- Excel 导入：pandas、openpyxl。
- 二维码：qrcode 或同类成熟库。
- 存证机制：SHA-256 + 本地哈希链优先；有余力再接 Hardhat/Ganache 或通过独立适配层接入测试链。

## 6. 数据一致性硬约束

所有代码必须围绕统一主键体系设计，避免演示时串不起来：

- `certificate_id`：系统内部证书主键。
- `certificate_no`：对外展示的证书编号。
- `student_id`：学生主键。
- `batch_id`：批量颁发批次。
- `pdf_file_id` 或 `pdf_path`：证书文件位置。
- `certificate_hash`：证书正文或 PDF 的 SHA-256 指纹。
- `receipt_id`：哈希链或测试链回执。
- `status`：有效、撤销、补发、过期等状态。
- `credential_type`：可信凭证类型，当前可默认为证书，未来可扩展为竞赛、科研、任职等凭证。
- `root_id`：可选字段，用于未来关联 Merkle Root 或批量承诺根。

证书编号、PDF、二维码链接、哈希、回执、撤销/补发记录必须能互相追踪。

如果引入 `CredentialItem` 抽象，必须保持向下兼容：证书仍然是第一类对象，不能为了未来简历扩展牺牲当前发证、验真、撤销闭环。

## 7. 开发优先级

1. 业务闭环优先：发证、下载、扫码、验真、撤销。
2. 可信机制第二：SHA-256、哈希链、回执、审计日志。
3. 演示体验第三：仪表盘、统计图、页面美化。
4. 提高项最后：Merkle Root、本地测试链、DID/VC、多机构、可信简历选择性披露。

## 8. 文档要求

课程提交文档需围绕真实实现编写，不能只写概念：

- 需求规格说明书：角色、用例、业务流程、功能需求、非功能需求、安全边界。
- 概要设计说明书：系统架构、模块划分、数据库、部署结构、存证机制。
- 详细设计说明书：证书生成、哈希计算、二维码验真、撤销补发、审计日志、异常流程。
- 测试计划：功能测试、异常测试、权限测试、篡改 PDF 验真失败、撤销后再次验真。

文档可以增加“后续扩展为诚信简历”的章节，但必须明确：当前实现完成证书类可信凭证闭环，诚信简历是基于多类 `CredentialItem` 的后续扩展。

## 9. 验收演示脚本

答辩演示必须提前准备固定数据和固定流程：

1. 教师创建“暑期实训结业证书”模板。
2. 导入 5-10 名学生 Excel 名单。
3. 批量生成证书 PDF、二维码和哈希。
4. 管理员完成本地哈希链存证。
5. 学生端下载某张证书。
6. 第三方扫码验真，显示“有效、哈希一致、回执存在”。
7. 手动修改 PDF 后上传或校验，显示“哈希不一致”。
8. 管理员撤销一张证书。
9. 第三方再次扫码，显示“证书已撤销”。
10. 可选加分演示：学生选择 2-3 张已发证书生成“可信凭证分享页”，说明这是诚信简历的最小雏形，不要求完整实现创三。

## 10. 编码与变更规则

- 优先小步提交，不做无关重构。
- 新增模块前先确认是否服务最小闭环。
- 不把密钥、真实学生隐私、账号 token 写入仓库。
- 测试数据必须使用模拟姓名、模拟学号、模拟机构。
- 每次重要改动后更新 `docs/项目规划/开发目标收束.md` 或对应设计文档。
- 代码完成后至少运行对应的启动、构建、单元测试或手动验收脚本，并记录结果。
- 如果新增 `CredentialItem`、Merkle Root、选择性披露相关设计，必须先确认不影响证书主流程。

## 11. AI 使用边界

可以用 Codex 辅助生成代码、表结构、接口、测试和文档，但不能让 AI 直接扩大范围。遇到以下情况先回到最小闭环：

- AI 建议接入复杂真链。
- AI 建议上 DID/VC 完整体系。
- AI 建议重构成微服务。
- AI 生成的页面很多但主流程跑不通。
- AI 生成的数据模型无法串起证书、PDF、哈希、二维码和撤销状态。
- AI 把项目改成完整诚信简历系统，导致证书存证主线失焦。

## 12. 老师课堂指导整理：测试链、哈希存证与项目边界

以下内容来自老师对本项目的课堂指导整理，后续方案、开发和答辩口径应优先与本节保持一致。

### 核心判断

- 本项目不需要一开始接入真实以太坊主网。主网由全球节点共同维护，写入和执行都需要真实 Gas 费用，课程实训阶段成本和风险都偏高。
- 测试链在功能上接近主网，但使用测试币，适合开发、调试和教学演示。测试币可通过测试网水龙头申请，也可由已有账户转入。
- 证书不应整份 PDF、图片或大文件直接上链。更合理的方式是链下保存证书文件，链上或本地哈希链只保存证书哈希、证书编号、机构、时间戳、状态和回执等关键数据。
- 证书哈希建议使用 SHA-256。无论证书文件多大，最终都映射为固定长度指纹，便于低成本存证和验真。
- 哈希一致只能证明某个证书文件版本在存证后未被篡改，不能单独证明证书天然真实；当前是否有效还必须结合系统状态表、撤销记录、补发记录和机构授权流程判断。
- 智能合约不是当前阶段的核心难点。若使用测试链，先做到关键哈希数据能写入、能查询交易回执、能支撑验真页面展示即可。

### 推荐实现路线

1. 优先完成证书业务闭环：模板配置、Excel 导入、批量生成证书编号、PDF、二维码和下载链接。
2. 对证书正文或 PDF 计算 SHA-256 哈希，并保存到业务数据库。
3. 将证书哈希、证书编号、机构、时间戳、状态等关键字段写入本地哈希链或以太坊测试链，生成 `receipt_id`。
4. 第三方扫码验真时，展示证书基础信息、哈希校验结果、当前状态和存证回执。
5. 管理员撤销或补发证书时，必须更新状态表并保留审计日志；再次验真时应明确展示“已撤销”或“已补发”。
6. 如果测试链、钱包或测试币申请短期调不通，立即降级为本地哈希链或模拟链回执，先保证演示闭环稳定。

### 测试链工具链口径

- 可以安排 1-2 名同学专门研究 MetaMask 小狐狸钱包、测试链网络配置、测试币水龙头、交易发送和回执查询。
- 钱包私钥、助记词、API token、真实账户信息不得写入代码、文档、日志或提交记录。
- 测试链只是教学演示环境，不能在文档或答辩中等同于生产级主网部署。
- 若使用 Hardhat、Ganache 或其他本地链，也应统一解释为“教学版可信存证模拟”，重点是流程可验证，而不是宣称已经接入真实公链。

### 与创新项目 3 的关系

- 本项目可以吸收“大学生诚信简历系统”的思想，但主线仍然是创新项目 1：证书存证与验真。
- 当前实现应优先支持单张证书的生成、哈希、存证、查询、验真、撤销和补发。
- Merkle Tree、选择性披露、多凭证可信简历只作为后续扩展或加分叙事，不应影响证书主流程稳定。
- Merkle Proof 要可信，必须锚定到可信 Root，例如链上回执、本地哈希链回执或学校签名 Root；否则只能算展示页，不能算可信证明。
- 旧 Proof 只能证明“曾被纳入某次承诺”，不能单独证明“当前仍有效”；当前有效性必须查询状态表、撤销记录或最新承诺 Root。

### 团队分工建议

- 区块链组：负责测试链、MetaMask、测试币申请、哈希上链、交易回执查询和降级方案。
- 后端组：负责证书数据模型、Excel 导入、批量生成、哈希计算、状态管理、撤销补发和审计日志。
- 前端组：负责教师端、学生端、第三方扫码验真页和演示页面体验。
- 文档组：负责需求规格说明书、概要设计、详细设计、测试计划和答辩讲稿。
- 测试组：负责正常验真、篡改 PDF 验真失败、撤销后再次验真、补发后追踪等演示用例。

### 答辩推荐表述

本系统没有把证书 PDF 原文件直接存到链上，因为链上存储成本高且没有必要。系统采用“链下证书文件 + 链上/本地哈希回执 + 系统状态表”的方式：证书生成后计算 SHA-256 哈希，并将哈希、证书编号、机构和时间戳写入测试链或本地哈希链。第三方验证时，系统重新计算证书文件哈希并与存证记录比对；若一致，说明该版本证书自存证后未被篡改；若不一致，说明文件内容发生变化。同时系统还会查询证书状态表，判断证书当前是否有效、撤销、补发或过期。

### 必须避免的错误表述

- 不要说“区块链保证证书绝对真实”，应说“证明某个证书文件或内容哈希在存证后未被篡改”。
- 不要说“测试链等同真实主网”，应说“测试链适合教学演示，主网部署是后续生产化方向”。
- 不要说“必须实现复杂智能合约”，应说“当前核心是业务闭环可验证，合约或链上写入先保持最小可用”。
- 不要说“Merkle Tree 实现后就天然可信”，应说“Merkle Proof 需要可信 Root、链上回执或学校签名 Root 锚定”。

## 13. 2026-07-23 当前执行基线与线上协作规则

以下规则以本轮已验证的线上部署为准，用于约束后续联调、文档和合并，不替代历史记录。

### 当前基线

- 集成分支为 `dev`。2026-07-23 的线上功能 release 源码为 `26e9324`，随后 `dev` 的文档记录继续更新；`main` 尚不是当前部署或联调基线。
- `rsy` 服务器当前 release 为 `20260723-022737-26e9324`。Docker Compose 的 `backend`、`db`、`web` 均健康；Web 仅监听宿主机 `127.0.0.1:18080`，Caddy 负责公网 HTTPS。
- 公共验真入口为 `https://verify.lotusrain.net/public/verify`，协作登录入口为 `https://verify.lotusrain.net/login`。2026-07-23 已验证 HTTPS、学生端和带模拟学号的证书查询 `200`、未传学号 `422`、未登录管理接口 `401`，以及文档、OpenAPI 和健康接口的公网 `404` 边界；学生页仅为临时演示，不是正式登录。
- `admin.verify.lotusrain.net` 的服务器端 DNS 尚未就绪，因此当前仅临时使用同域受限入口。Caddy 放行登录、注册、指定管理页面、`/api/auth/*`、`/api/admin/*`，以及本次批准的临时学生页和当前证书查询路径；后端仍必须独立执行 JWT 与角色鉴权，不能把 Caddy 白名单视为权限控制。

### 账号与权限边界

- 首个管理员只能在服务器交互式创建；不得生成、打印、提交或通过聊天传递管理员密码。
- 管理员通过一次性教师邀请链接创建团队账号。邀请码原文只在创建时展示一次，应通过可信渠道发送；链接使用 URL fragment 承载 token，不能改回查询参数。
- `ADMIN` 可管理账号和创建教师邀请；`TEACHER` 可处理项目、学生、模板、批次和签发；`AUDITOR` 仅可读取存证与审计数据。任何角色判断必须由后端执行，前端菜单隐藏不是安全边界。
- 现有学生端仍依赖 `student_no` 演示参数，不得写成正式身份认证。2026-07-23 经项目负责人明确批准，Caddy 临时放行 `/student/*` 与当前证书查询接口 `/api/student/certificates` 及其子路径供模拟数据演示；不得导入真实学生资料，且完成演示后应恢复拦截或先实现学生账号绑定与 JWT。当前“开放注册”仍仅指管理员受控发出的教师邀请注册。

### 当前收口顺序

1. 使用真实教师邀请账号在服务器完成：项目/模板 -> Excel 导入 -> 批次 -> PDF/二维码 -> 存证 -> 公共验真 -> 篡改复验 -> 撤销/补发。
2. 对每个环节保留不含隐私和 token 的截图、证书编号、状态、回执和审计证据；不提交生成 PDF、二维码、数据库或日志。
3. 回归未登录 `401`、教师越权、邀请重复使用失败和禁用账号令牌失效；若已有受控 `AUDITOR` 账号，再回归其越权 `403`，不得通过手工改库创建账号。同步验证公网 Caddy 仅开放公共验真、协作管理和临时学生演示路径，文档/健康/OpenAPI 等边界未扩大。
4. 仅在上述线上全流程、后端测试、前端构建、课程材料和敏感文件检查均通过后，才可创建 `dev -> main` 稳定版 PR。
