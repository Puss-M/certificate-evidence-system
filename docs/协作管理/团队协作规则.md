# 团队协作规则

本文档用于约束 5 人小组在两周实训中的协作方式。目标是保证项目每天可集成、可演示、可回滚，避免 Git 冲突和功能范围膨胀。

## 1. 角色分工

| 角色 | 主要负责人 | 核心职责 | 交付物 |
| --- | --- | --- | --- |
| 1 号：总体设计、系统集成与部署 | 项目负责人 | 架构、接口、Git、联调、部署、文档整合、答辩 PPT | 架构图、接口规范、数据库设计、联调记录、部署说明 |
| 2 号：管理员端前端与链上管理展示 | 前端负责人 A | 管理端页面、路由权限、模板/学生/批次/证书/日志页面 | Vue 页面、接口调用说明、运行截图 |
| 3 号：学生端、公共验真端与验真测试 | 前端/测试负责人 | 学生中心、证书详情、二维码、扫码验真、上传 PDF 验真、异常测试 | 验真页、测试用例、截图/录屏 |
| 4 号：后端基础业务与区块链业务对接 | 后端负责人 | FastAPI、MySQL、登录、JWT、业务接口、状态同步 | 后端接口、数据库脚本、接口测试结果 |
| 5 号：证书生成、智能合约与可信存证 | 存证负责人 | 证书编号、PDF、二维码、SHA-256、本地哈希链、回执 | 生成模块、存证模块、示例证书、回执记录 |

## 2. 总体协作原则

1. 主线优先：先完成“导入名单 -> 生成证书 -> 哈希存证 -> 扫码验真 -> 撤销”的最小闭环。
2. 接口先行：字段名和接口路径确定后，不随意修改；必须修改时先通知全组。
3. 小步集成：每天至少集成一次可运行版本，不把大量未验证代码一次性合入。
4. 本地哈希链兜底：FISCO BCOS、智能合约和链上展示作为加分项，不能阻塞主流程。
5. 模拟数据优先：不得提交真实学生隐私、真实账号密码、API token 或密钥。

## 3. Git 使用规则

### 3.1 分支规则

| 分支 | 用途 | 管理规则 |
| --- | --- | --- |
| `main` | 稳定可演示版本 | 只接收通过完整回归的 `dev → main` PR，由 1 号负责人合并 |
| `dev` | 日常集成版本 | 所有功能、修复和文档 PR 的统一目标分支，不允许绕过 |
| `feature/admin-frontend` | 管理端前端 | 2 号使用 |
| `feature/student-verify` | 学生端和验真端 | 3 号使用 |
| `feature/backend-api` | 后端接口 | 4 号使用 |
| `feature/certificate-evidence` | 证书生成和存证 | 5 号使用 |
| `docs/project-docs` | 文档和答辩材料 | 文档负责人或 1 号使用 |

### 3.2 提交规则

提交信息必须能看懂，推荐格式：

```text
完成管理员证书列表页面
新增证书批次接口
修复撤销后验真状态未更新
补充演示验收脚本
```

不推荐使用：

```text
update
fix
111
改了一点
```

### 3.3 合并规则

所有成员提交 PR 时必须遵守：

1. `feature/*`、`fix/*`、`docs/*` 和个人开发分支的 PR，目标分支统一选择 `dev`。
2. 不允许功能分支直接向 `main` 提交 PR；如发现目标分支错误，先修改 PR base 或重新开 PR，不能直接合并后再补救。
3. `dev` 完成后端测试、前端构建、真实环境联调和 1 号验收后，才允许创建 `dev → main` PR。
4. `main` 只保存稳定可演示版本，不承担日常冲突解决和功能集成。

合并到 `dev` 前必须满足：

1. 项目能启动或模块能独立运行。
2. 不修改与自己模块无关的大量文件。
3. 不引入密钥、真实隐私数据、构建产物和临时文件。
4. 变更说明包含：改了什么、怎么运行、是否影响接口。
5. 至少提供一张运行截图或接口测试结果。

合并 `dev → main` 前还必须满足：

1. 后端完整测试和前端生产构建通过。
2. MySQL、FastAPI、Vue 在统一环境完成主流程回归。
3. 导入、签发、存证、下载、验真、篡改、撤销和补发均有证据。
4. 1 号确认无敏感文件、未验证代码和未处理冲突。

## 4. 每日交付格式

每位成员每天向 1 号负责人提交以下内容：

```text
【今日完成】
- 完成了什么功能/文档/测试

【涉及文件】
- 文件或目录路径

【运行方式】
- 如何启动或如何查看结果

【截图/结果】
- 页面截图、接口返回、生成文件或测试结果

【需要联调】
- 需要谁配合、依赖哪个接口或字段

【风险】
- 未完成、报错、接口不确定或明天要处理的问题
```

## 5. 冲突处理

1. 先保主流程，不为加分功能打断主线。
2. 前后端接口不一致时，以 `docs/协作管理/接口规范.md` 为准。
3. 数据库字段不一致时，以 `docs/协作管理/数据库设计.md` 为准。
4. 代码冲突无法快速解决时，由 1 号负责人决定保留哪一版。
5. FISCO BCOS 接入失败时，立即切回本地哈希链，不影响演示。

## 6. 敏感信息禁止提交

禁止提交：

- `.env`、数据库密码、私钥、证书文件、token。
- 真实学生姓名、真实学号、身份证号、手机号。
- 本地数据库文件、日志大文件、构建产物。
- 微信临时文件、个人桌面路径中的无关资料。

演示数据统一使用模拟姓名、模拟学号和模拟机构。
