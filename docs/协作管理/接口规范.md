# 接口规范

本文档用于统一前端、后端、证书生成模块和存证模块之间的接口口径。实际接口可以随技术实现微调，但核心字段含义必须保持一致。

## 1. 通用约定

### 1.1 基础路径

```text
/api
```

### 1.2 通用返回格式

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

### 1.3 状态码约定

| code | 含义 |
| --- | --- |
| 0 | 成功 |
| 400 | 请求参数错误 |
| 401 | 未登录或 token 失效 |
| 403 | 无权限 |
| 404 | 数据不存在 |
| 409 | 状态冲突，例如重复撤销 |
| 422 | 请求体或字段格式校验失败 |
| 500 | 系统异常 |

## 2. 核心字段字典

| 字段 | 类型 | 含义 | 说明 |
| --- | --- | --- | --- |
| `certificate_id` | Long/String | 系统内部证书主键 | 后端内部使用 |
| `certificate_no` | String | 对外展示证书编号 | 二维码和验真优先使用 |
| `student_id` | Long/String | 学生主键 | 关联学生记录 |
| `student_no` | String | 学号 | 学生账号用户名；仅管理员开通时使用，学生端不接受此身份参数 |
| `student_name` | String | 学生姓名 | 关联学生档案；学生端从 JWT 绑定的 `student_id` 查询 |
| `batch_id` | Long/String | 证书批次主键 | 批量发证使用 |
| `template_id` | Long/String | 证书模板主键 | PDF 生成使用 |
| `pdf_path` | String | PDF 文件路径 | 保存相对路径或文件 ID |
| `certificate_hash` | String | 证书 PDF 的 SHA-256 | 对最终 PDF 计算 |
| `qr_code_path` | String | 二维码图片路径 | 可选 |
| `verify_url` | String | 验真链接 | 二维码指向该链接 |
| `receipt_id` | String | 存证回执编号 | 本地哈希链或测试链返回 |
| `status` | String | 证书状态 | `VALID`、`REVOKED`、`REISSUED`、`EXPIRED` |
| `evidence_status` | String | 管理端存证展示状态 | `CONFIRMED` 表示已有回执，`PENDING` 表示待存证 |
| `credential_type` | String | 凭证类型 | 当前默认 `CERTIFICATE` |
| `root_id` | String | 关联的 Root 业务编号 | 对外返回 `credential_roots.root_no`；表内自增主键不暴露 |
| `previous_certificate_no` | String | 补发新证书指向旧证书编号 | 补发时必填 |
| `merkle_root` | String | 批次 Merkle Root | P2 本地 Root/Proof 接口返回 |
| `merkle_proof` | Array&lt;Object&gt; | 叶子到 Root 的兄弟节点路径 | P2 Proof 接口返回 |
| `root_chain_hash` | String | Root 链哈希 | 当前接口字段名为 `current_root_hash` |

## 3. 认证接口

### 3.1 登录

```http
POST /api/auth/login
```

请求：

```json
{
    "username": "<管理员、教师、审计员或学生用户名>",
  "password": "<账号密码>"
}
```

响应：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "token": "jwt-token",
    "role": "ADMIN",
    "display_name": "系统管理员",
    "student_id": null,
    "must_change_password": false
  }
}
```

线上环境不提供固定演示账号。管理员密码通过服务器交互式初始化，教师账号只能由管理员创建一次性邀请后注册；学生账号由管理员为已导入学生批量开通，用户名为学号。

### 3.2 查询当前会话与退出

```http
GET /api/auth/me
POST /api/auth/logout
Authorization: Bearer <token>
```

- `GET /api/auth/me` 返回当前账号的 `user_id`、`username`、`display_name`、`role`、`student_id`、`must_change_password` 和 `is_active`。
- `POST /api/auth/logout` 撤销当前会话；之后同一 token 调用管理接口必须返回 `401`。

### 3.3 修改密码

```http
POST /api/auth/change-password
Authorization: Bearer <token>
```

请求字段：`current_password`、`new_password`（至少 12 位）。学生使用初始密码登录后，必须先调用此接口；在改密完成前访问学生证书接口返回 `403`。

### 3.4 受邀教师注册

```http
POST /api/auth/register/invitation
```

请求：

```json
{
  "invitation_token": "<仅从邀请链接 fragment 读取的 token>",
  "username": "teacher.demo",
  "display_name": "模拟教师",
  "password": "至少12位的本地测试密码"
}
```

- 邀请 token 在前端仅通过 URL fragment 传递，注册成功后立即清除 fragment；不得改为查询参数或写入截图、日志、文档。
- 邀请只能注册 `TEACHER`，有效期为 1-168 小时，使用、过期或无效时返回 `400`；用户名重复返回 `409`。

### 3.5 管理员邀请与账号状态

以下接口均要求 `ADMIN` 角色：

```http
POST  /api/admin/invitations
GET   /api/admin/users
PATCH /api/admin/users/{user_id}/status
```

创建邀请请求示例：

```json
{
  "expires_in_hours": 48
}
```

创建响应中的 `invitation_token` 只展示一次，管理员负责生成前端注册链接并通过可信渠道发送。禁用账号会撤销该账号所有未过期会话；不能禁用当前登录的管理员自身，冲突时返回 `409`。

## 4. 管理端接口

### 4.1 创建证书模板

```http
POST /api/admin/templates
```

请求：

```json
{
  "template_name": "暑期实训结业证书",
  "institution_name": "示范学院",
  "content_config": {
    "project_id": 1,
    "project_name": "软件开发实训",
    "issue_year": "2026"
  }
}
```

`content_config.project_id` 是模板绑定的真实项目主键。后端会校验项目存在，并以项目表中的名称覆盖 `project_name`；项目改名时会同步模板显示名称，模板仍绑定该项目时项目删除返回 `409`。创建批次时，所选项目必须与模板绑定项目一致。

### 4.2 导入学生名单

```http
POST /api/admin/students/import
Content-Type: multipart/form-data
```

字段：

| 字段 | 含义 |
| --- | --- |
| `file` | Excel 文件 |
| `batch_name` | 批次名称 |
| `template_id` | 模板 ID |

Excel 首行支持以下中英文表头，可混用：

| 字段 | 是否必填 | 约束 |
| --- | --- | --- |
| `学号` / `student_no` | 是 | 同一文件及数据库内不得重复 |
| `姓名` / `student_name` | 是 | 不得为空 |
| `学院` / `college` | 否 | 最长 100 个字符 |
| `班级` / `class_name` | 否 | 学生班级 |
| `专业` / `major_name` | 否 | 学生专业 |

学院字段必须原样持久化并在管理端学生列表、新增和编辑接口中返回，不能继续使用固定“示范学院”占位值。超长学院名称按行记入 `failures`，不得导致整份 Excel 回滚。

### 4.2.1 开通与重置学生账号

以下接口仅要求 `ADMIN` JWT；接口会返回一次性初始密码，教师不应获取或分发该凭据：

```http
POST /api/admin/students/accounts/provision
POST /api/admin/students/{student_id}/account/reset-password
```

批量开通请求可指定 `student_ids` 数组；空数组表示开通全部尚未绑定账号的学生。成功响应的 `created` 中包含 `student_id`、`student_no`、`student_name` 和仅本次响应可见的 `initial_password`；`skipped` 说明已开通、学号冲突或学生不存在的条目。数据库不得保存该明文密码。

重置接口为已开通学生生成一次性新密码，并立即撤销该学生现有会话；学生下次登录后必须修改密码。学生完成首次改密时，也会撤销同一账号的其他有效会话。

### 4.3 创建证书批次

```http
POST /api/admin/batches
```

请求：

```json
{
  "template_id": 1
}
```

管理端按已绑定实训项目的模板创建空批次。后端从模板的 `content_config.project_id` 推导项目和项目名称，并自动生成批次名称与批次编号；模板不存在返回 `404`，未绑定项目或其项目失效返回 `409`。

空请求或未提供模板且未提供历史批次名称的请求返回 `422`。

为兼容历史调用方，仍可额外传 `batch_name`、`project_id`、`project_name` 和 `student_ids`。显式 `project_id` 必须与模板绑定项目一致；学生名单以生成接口中传入的 `student_ids` 为准。

### 4.4 批量生成证书

```http
POST /api/admin/batches/{batch_id}/generate
```

管理员前端调用时传入待签发学生与日期：

```json
{
  "student_ids": [1, 2, 3],
  "project_id": 1,
  "template_id": 1,
  "issue_date": "2026-07-15"
}
```

响应：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "batch_id": 1,
    "generated_count": 5,
    "failed_count": 0,
    "failures": []
  }
}
```

请求中显式传入的 `template_id` 必须对应已存在的模板；模板不存在时返回 `404`，不得静默改用其他模板。学生名单、项目、模板会在签发前校验并写回批次，以便批次详情、证书和存证记录保持可追溯。只有未提供 `template_id` 时，系统才可使用默认模板配置。

#### 4.4.1 带导师签章批量生成

```http
POST /api/admin/batches/{batch_id}/generate-with-signature
Content-Type: multipart/form-data
```

字段为 `student_ids`（JSON 整数数组字符串）、`issue_date`、`template_id`、`project_id` 和 `mentor_signature`（PNG/JPEG 文件）。仅当模板 `fields` 含 `mentor_signature` 时可调用；服务端将图片居中裁剪为 4:1，仅用于本次生成且不落库。签章图片最大 20,000,000 像素；无效、空或像素过大的图片返回 `400` 或 `422`。

#### 4.4.2 模板真实 PDF 预览

```http
GET /api/admin/templates/{template_id}/preview
```

返回 `application/pdf` 二进制，使用固定示例学生和证书编号走与正式发证相同的 PDF 渲染逻辑；不创建证书、不会占用编号或写入存证回执。模板不存在返回 `404`。

### 4.5 批量存证

```http
POST /api/admin/batches/{batch_id}/evidence
```

响应：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "batch_id": 1,
    "success_count": 5,
    "receipt_ids": ["RCP-20260713-0001"],
    "evidenced": 5,
    "newly_evidenced": 5
  }
}
```

### 4.6 撤销证书

```http
POST /api/admin/certificates/{certificate_identifier}/revoke
```

请求：

```json
{
  "reason": "证书信息有误，需重新签发"
}
```

### 4.6.1 管理端下载证书

```http
GET /api/admin/certificates/{certificate_no}/download
Authorization: Bearer <ADMIN_OR_TEACHER_JWT>
```

管理端下载必须携带管理员或教师 JWT。公共验真接口不提供 PDF 下载；学生下载仅使用绑定当前 `student_id` 的 `/api/student/certificates/{certificate_no}/download`。

### 4.7 补发证书

```http
POST /api/admin/certificates/{certificate_id}/reissue
```

请求：

```json
{
  "reason": "原证书信息有误，重新签发",
  "issue_date": "2026-07-15"
}
```

仅状态为 `REVOKED` 的旧证书可以补发。系统为指定旧证书生成新的证书编号、PDF、二维码、SHA-256 哈希和存证回执，将旧证书状态更新为 `REISSUED`，并通过 `previous_certificate_no` 保留新旧证书关系。

响应 `data` 包含 `old_certificate` 和 `new_certificate` 两个证书对象。旧证书不存在时返回 `404`；旧证书未撤销或已经补发时返回 `409`；请求字段不符合约束时返回 `422`。

### 4.8 生成 Merkle Root（P2 本地实现）

```http
POST /api/admin/batches/{batch_id}/merkle-root
```

批次内证书必须已经生成 SHA-256 和本地存证回执。同一 `batch_id` 只能生成一次 Root；重复请求返回 `409`，批次不存在返回 `404`，证书或回执不完整返回 `400`。

响应关键字段：`batch_id`、`root_id`、`root_no`、`merkle_root`、`leaf_order_rule`、`odd_leaf_rule`、`previous_root_hash`、`current_root_hash`、`leaf_count`、`tx_hash`。其中 `root_id` 与 `root_no` 当前都返回 Root 业务编号，保留双字段是为了兼容已冻结契约。

该接口先完成本地计算、落库和 Root 哈希链关联；若配置了允许的本地测试链、后端测试钱包和合约地址，会额外尝试写入批次 Root。写链成功时 `tx_hash` 为交易哈希，未配置或失败时为 `null`，且不影响本地 Root 结果。代码默认只允许链 ID `1337`、`31337`，不得配置真实主网或产生真实费用。

### 4.9 查询批次 Merkle Root

```http
GET /api/admin/batches/{batch_id}/merkle-root
```

返回该批次已经持久化的 Root，响应字段与 4.8 的 POST 接口一致：`batch_id`、`root_id`、`root_no`、`merkle_root`、`previous_root_hash`、`current_root_hash`、`leaf_count`、`leaf_order_rule`、`odd_leaf_rule`、`tx_hash`。该接口只查询本地数据库，不重新计算 Root，也不访问测试链；尚未生成 Root 时返回 `404`。

## 5. 学生端接口

以下 `/api/student/certificates*` 路径要求 `STUDENT` JWT。学生身份由登录账号的唯一 `student_id` 绑定确定，客户端不得传入、缓存或修改 `student_no` 以选择查询对象。初始密码状态下访问这些接口返回 `403`，学生需先完成改密。

### 5.1 我的证书列表

```http
GET /api/student/certificates
```

请求头：`Authorization: Bearer <student-token>`。

响应字段：

| 字段 | 含义 |
| --- | --- |
| `certificate_no` | 证书编号 |
| `project_name` | 项目名称 |
| `issue_time` | 签发时间 |
| `status` | 当前状态 |
| `verify_url` | 验真链接 |

### 5.2 证书详情

```http
GET /api/student/certificates/{certificate_no}
```

请求头：`Authorization: Bearer <student-token>`。证书不属于当前登录学生时返回 `404`，不泄露他人证书详情。

### 5.3 下载证书 PDF

```http
GET /api/student/certificates/{certificate_no}/download
```

请求头：`Authorization: Bearer <student-token>`。仅在证书归属匹配且 PDF 文件存在时返回 `application/pdf`。

### 5.4 获取证书二维码

```http
GET /api/student/certificates/{certificate_no}/qrcode
```

请求头：`Authorization: Bearer <student-token>`。仅在证书归属匹配且二维码文件存在时返回 `image/png`。

新生成证书的 `verify_url` 和二维码默认指向前端页面 `/public/verify/{certificate_no}`。若修改 `PUBLIC_VERIFY_BASE_URL`，必须重新生成证书和二维码；旧文件不会自动更新。

## 6. 公共验真接口

### 6.1 编号验真或扫码验真

```http
GET /api/verification/{certificate_no}
```

响应：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "certificate_no": "CERT-20260713-0001",
    "student_name": "张三",
    "project_name": "软件开发实训",
    "institution_name": "计算机学院",
    "status": "VALID",
    "certificate_hash": "sha256-hex",
    "receipt_id": "RCP-20260713-0001",
    "receipt_exists": true,
    "hash_match": true,
    "verify_result": "PASS",
    "verify_message": "证书有效，哈希一致，存证回执存在",
    "result": "PASS",
    "message": "证书有效，哈希一致，存证回执存在"
  }
}
```

说明：编号/扫码验真的 `hash_match` 是后端重新计算服务端留存 PDF 后与 `certificate_hash` 比对的结果；它不代表第三方手中任意文件都一致。`result`、`message` 是现有调用方的兼容字段，新调用方优先使用 `verify_result`、`verify_message`。证书生成时会把模板的签发机构快照持久化到证书记录，后续修改模板不会改变已签发证书的 `institution_name`；仅缺少历史数据且无法回填时该字段可能为 `null`。

### 6.2 上传 PDF 复验

```http
POST /api/verification/{certificate_no}/file
Content-Type: multipart/form-data
```

字段：

| 字段 | 含义 |
| --- | --- |
| `file` | 待复验 PDF |

响应关键字段：

| 字段 | 含义 |
| --- | --- |
| `uploaded_hash` | 上传文件计算出的 SHA-256 |
| `stored_hash` | 系统存证哈希 |
| `hash_match` | 是否一致 |
| `verify_result` | 验真结果，例如 `PASS`、`HASH_MISMATCH` |
| `result` | 兼容字段，与 `verify_result` 相同 |

### 6.3 Merkle Proof（P2 本地实现）

```http
GET /api/public/verify/{certificate_no}/merkle-proof
```

兼容路径：`GET /api/verification/{certificate_no}/merkle-proof`。

响应关键字段：`certificate_no`、`certificate_hash`、`leaf_index`、`leaf_order_rule`、`odd_leaf_rule`、`root_id`、`root_no`、`merkle_root`、`merkle_proof`、`proof_valid`。兼容字段 `proof` 与 `merkle_proof` 相同，`verified` 与 `proof_valid` 相同。

`proof_valid=true` 只证明该证书哈希属于当前 Root 且 Proof 可重算通过，不代表证书当前仍有效，也不能单独证明 Root 已写入测试链；链上锚定状态必须结合 `tx_hash` 和链上记录查询，有效性仍需查询证书状态和撤销记录。

## 7. 存证服务内部接口

业务后端调用存证模块时，统一使用稳定接口，底层可以是本地哈希链，也可以是 FISCO BCOS。

### 7.1 保存存证

```text
saveEvidence(certificateNo, certificateHash, metadata) -> EvidenceReceipt
```

返回：

```json
{
  "receipt_id": "RCP-20260713-0001",
  "evidence_type": "LOCAL_HASH_CHAIN",
  "block_height": 1,
  "tx_hash": null,
  "evidence_time": "2026-07-13 12:00:00",
  "success": true
}
```

### 7.2 查询存证

```text
getEvidence(receiptId) -> EvidenceReceipt
```

## 8. 验真结果枚举

| 结果 | 含义 | 页面提示 |
| --- | --- | --- |
| `PASS` | 验真通过 | 证书有效，哈希一致，存证回执存在 |
| `REVOKED` | 已撤销 | 证书已撤销，不再有效 |
| `REISSUED` | 已补发 | 旧证书已补发，请查看新证书 |
| `HASH_MISMATCH` | 哈希不一致 | 上传文件与存证版本不一致，可能被篡改 |
| `NOT_FOUND` | 不存在 | 未查询到该证书编号 |
| `NO_RECEIPT` | 无回执 | 证书存在，但未完成存证 |
| `SYSTEM_ERROR` | 系统异常 | 验真服务暂不可用 |

## 9. 闭环强制约束

### 9.1 撤销与补发一致性

1. 撤销旧证书时，必须将旧证书 `status` 更新为 `REVOKED`，并记录撤销原因、撤销人和撤销时间。
2. 补发证书时，不允许复用旧证书编号、旧 PDF、旧哈希或旧回执。
3. 新证书必须重新生成：`certificate_no`、`pdf_path`、`qr_code_path`、`verify_url`、`certificate_hash`、`receipt_id`。
4. 新旧证书必须保留关联：旧证书记录 `new_certificate_no` 或新证书记录 `previous_certificate_no`。
5. 公共验真旧二维码时，应显示旧证书已撤销；公共验真新二维码时，应显示新证书当前状态。

### 9.2 公共验真接口强制返回字段

公共验真接口不得只返回“有效/无效”，必须返回以下字段，方便前端展示失败原因：

| 字段 | 是否必填 | 含义 |
| --- | --- | --- |
| `certificate_no` | 是 | 证书编号 |
| `status` | 证书存在时必填 | 当前状态 |
| `certificate_hash` | 已生成哈希时必填 | 系统保存的证书哈希 |
| `receipt_id` | 否 | 存证回执编号 |
| `receipt_exists` | 是 | 对应回执记录是否真实存在 |
| `hash_match` | 是 | 编号验真时校验留存 PDF，上传验真时校验上传文件；同时要求回执哈希一致 |
| `verify_result` | 是 | 验真结果枚举 |
| `verify_message` | 是 | 面向用户的结果说明 |
| `result` | 是 | 兼容字段，与 `verify_result` 相同 |
| `message` | 是 | 兼容字段，与 `verify_message` 相同 |
| `uploaded_hash` | 上传 PDF 时必填 | 上传文件的 SHA-256 |
| `stored_hash` | 已生成哈希时必填 | 系统保存的证书哈希 |
| `revocation_reason` | 撤销时必填 | 撤销原因 |
| `revoked_at` | 撤销时必填 | 撤销时间 |

### 9.3 权限边界

| 角色 | 可以做 | 不可以做 |
| --- | --- | --- |
| ADMIN | 创建教师邀请、管理账号、开通/重置学生账号、模板、导入、发证、存证、撤销、补发、查看日志 | 直接修改已存证哈希、禁用当前登录管理员 |
| TEACHER | 模板、导入、发证、存证、撤销和补发 | 创建邀请、管理账号、开通/重置学生账号、直接修改已存证哈希；当前前端不提供操作日志页 |
| AUDITOR | 只读查看 `/api/admin/evidence/*` 与 `/api/admin/audit-logs*` | 学生、项目、模板、批次、证书等其他管理数据及全部写操作 |
| 学生 | 查看和下载自己的证书、分享二维码 | 撤销证书、修改证书状态、查看他人证书 |
| 第三方验证者 | 通过编号/二维码/PDF 上传验真 | 下载未授权证书、查看后台日志 |

线上版本使用用户表、Argon2 密码哈希、带会话 `jti` 的 JWT 和后端角色鉴权。未登录或已撤销/禁用的 token 返回 `401`；角色不足返回 `403`。前端菜单权限只负责展示，不能代替后端校验。学生端使用 `STUDENT` JWT 的唯一 `student_id` 绑定，不接受 `student_no` 查询参数。

| 角色 | GET/HEAD/OPTIONS | POST/PUT/DELETE |
| --- | --- | --- |
| `ADMIN` | 允许 | 允许 |
| `TEACHER` | 允许 | 允许 |
| `AUDITOR` | 仅 `/api/admin/evidence/*` 与 `/api/admin/audit-logs*` 允许，其他管理路径为 `403` | `403` |
| 未登录或无效 token | `401` | `401` |

### 9.4 Merkle Root 一致性约束（P2 本地实现）

当前已实现本地 Root 计算、持久化和 Proof 验证，并支持将批次 Root 可选锚定到允许的本地测试链。测试链失败时自动降级为本地 Root；生产主网接入仍属于后续项。

1. `merkle-root` 接口只应在批次内所有证书都已生成哈希后调用一次，不随单张证书生成、撤销或补发而重复触发。
2. 撤销或补发证书时，不得修改任何已生成的 `credential_roots` 记录，`root_id`/`merkle_root` 一经写入即为历史事实，只能新增新批次的 Root，不能覆盖或删除旧 Root。
3. 验真接口如同时返回 Merkle Proof 结果和证书状态，前端必须分开展示，不能用其中一个覆盖另一个的结论（例如“内容验证通过但证书已撤销”必须两条信息都显示）。
4. 叶子排序固定为 `CERTIFICATE_NO_ASC`，即同一批次内按 `certificate_no` 升序排列；奇数叶固定使用 `DUPLICATE_LAST`，即复制当前层最后一个节点参与父节点计算。前后端、测试脚本和文档示例必须使用同一套规则。
