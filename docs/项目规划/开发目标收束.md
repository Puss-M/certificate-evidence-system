# 开发目标收束：基于区块链的实训证书与学业证明存证系统

## 1. 最终选题

本次暑期实训选定：

> 创新项目 1：基于区块链的实训证书与学业证明存证系统。

项目不以“做一个复杂区块链系统”为目标，而以“证书生命周期管理 + 哈希存证验真”为主线。两周内交付一个可运行、可演示、可测试、可写文档的最小可信闭环。

## 2. 一句话项目定义

为课程实训、竞赛、培训结业等场景提供证书批量生成、哈希存证、二维码验真、撤销补发和审计追踪能力，降低证书 PDF 被复制、篡改后难以验证的问题。

## 3. 核心用户与角色

| 角色 | 主要目标 | 核心操作 |
| --- | --- | --- |
| 课程教师 | 批量颁发课程证书 | 配置模板、导入名单、发起颁发 |
| 学院管理员 | 管理证书和存证状态 | 审核批次、存证、撤销、补发、查看日志 |
| 学生 | 获取并分享自己的证书 | 查询、下载 PDF、分享二维码 |
| 第三方验证者 | 判断证书是否有效 | 扫码验真、查看哈希和状态 |
| 系统管理员 | 维护系统基础数据 | 账号、机构、权限、日志管理 |

## 4. 两周最小闭环范围

### 必须完成

1. 证书模板管理：课程名称、项目名称、姓名、编号、成绩等级、导师签章等字段。
2. Excel 批量导入：导入学生名单并创建证书批次。
3. 批量生成证书：生成证书编号、PDF 文件和二维码。
4. 哈希指纹：对证书正文或 PDF 计算 SHA-256。
5. 本地存证：将证书哈希、颁发机构、时间戳、状态和回执写入本地哈希链或本地测试链。
6. 学生端：查询、下载、分享证书二维码。
7. 验真端：扫码后校验证书哈希、撤销状态和存证回执。
8. 撤销补发：支持状态变更并保留操作日志。

### 可以延后

1. Merkle Root 批量上链。
2. DID/VC 可信凭证钱包原型。
3. 多课程、多学院、多机构授权边界。
4. 审计报表和复杂统计看板。
5. 真实短信、邮件、电子签章或真实链上部署。

## 5. 核心业务流程

### 发证流程

1. 教师创建证书模板。
2. 教师导入学生 Excel 名单。
3. 系统创建证书批次。
4. 系统为每名学生生成证书编号。
5. 系统生成证书 PDF。
6. 系统计算证书 SHA-256 哈希。
7. 系统生成二维码验真链接。
8. 管理员确认批次并执行存证。
9. 学生查询并下载证书。

### 验真流程

1. 第三方扫描证书二维码。
2. 系统根据二维码中的证书编号或验真 token 查找证书。
3. 系统展示证书基础信息、当前状态、哈希指纹和存证回执。
4. 如支持文件上传复验，第三方上传 PDF 后系统重新计算哈希。
5. 哈希一致且状态有效，则显示验真通过；否则显示失败原因。

### 撤销补发流程

1. 管理员选择证书并填写撤销原因。
2. 系统将证书状态改为已撤销。
3. 系统记录撤销日志和操作者。
4. 验真端再次扫码时显示已撤销。
5. 如需补发，系统创建新证书编号、新 PDF、新哈希和新回执，并保留与旧证书的关联。

## 6. 核心数据对象

| 对象 | 用途 |
| --- | --- |
| `CertificateTemplate` | 证书模板配置 |
| `CertificateBatch` | 批量颁发批次 |
| `Student` | 学生信息，使用模拟数据 |
| `Certificate` | 单张证书主记录 |
| `CertificateHash` | 证书哈希指纹记录 |
| `ChainReceipt` | 哈希链或测试链回执 |
| `VerificationRecord` | 第三方验真记录 |
| `RevocationRecord` | 撤销、补发记录 |
| `Institution` | 颁发机构 |
| `RolePermission` | 角色和权限 |
| `AuditLog` | 操作审计日志 |

## 7. 推荐技术边界

### 稳妥主线

- 后端：FastAPI。
- 数据库：MySQL。
- 前端：Vue 3。
- PDF：reportlab、docxtpl 等 Python 生态稳定方案。
- Excel：openpyxl、pandas 等稳定方案。
- 二维码：qrcode 或同类成熟库。
- 存证：SHA-256 + 本地哈希链。

### 加分路线

- Hardhat/Ganache 本地测试链。
- 批量证书 Merkle Root。
- 简化版 DID/VC 概念展示。

## 8. 验收标准

项目完成时至少满足：

1. 能启动系统并完成完整演示。
2. 能导入一批学生并生成多张证书。
3. 每张证书都有编号、PDF、二维码、哈希和回执。
4. 扫码验真页面能显示有效、撤销、哈希不一致等状态。
5. 撤销后再次扫码能看到状态变化。
6. 有操作日志或审计记录可展示。
7. 有需求、概要、详细设计和测试文档。
8. 有固定演示数据和答辩脚本。

## 9. 主要风险与应对

| 风险 | 表现 | 应对 |
| --- | --- | --- |
| 过度追求真链 | 花太多时间调链，主流程不通 | 默认本地哈希链，真链只做提高项 |
| PDF 与哈希不一致 | 生成后修改、路径混乱、验真失败 | 固定哈希口径，优先对最终 PDF 计算哈希 |
| 二维码链路断裂 | 二维码找不到证书或环境地址变化 | 二维码使用稳定验真 token，演示前固定本机地址 |
| 撤销状态混乱 | 撤销后仍显示有效 | 状态字段统一由 `Certificate.status` 控制 |
| AI 生成范围过大 | 页面很多但主线不稳 | 每次新增功能先检查是否服务最小闭环 |
| 真实隐私风险 | 使用真实学生数据 | 全部使用模拟数据 |

## 13. 2026-07-15 接口契约收束记录

- 公共验真按接口规范补齐 `verify_result`、`verify_message`、`receipt_exists`、`hash_match`、`stored_hash`，并暂时保留 `result`、`message` 兼容字段。
- 编号/扫码验真重新计算服务端留存 PDF 的 SHA-256；上传复验计算用户上传文件的 SHA-256，两种校验口径分开说明。
- 404、422、500 等异常统一使用 `{code, message, data}` 响应结构。
- OpenAPI 同步声明统一错误模型和批量存证响应 Schema，避免生成客户端继续使用 FastAPI 默认的 `detail[]` 类型。
- Merkle Root / Proof 已完成本地 P2 实现：同批次 Root 唯一、固定 `CERTIFICATE_NO_ASC` 排序、奇数叶 `DUPLICATE_LAST`，并提供公开 Proof 查询；它仍不进入 P0 主链路。
- 当前 Root 仅保存在业务数据库并串成本地 Root 哈希链，尚未锚定 Hardhat、Ganache、FISCO BCOS 或其他测试链，不能称为“已经上链”。
- 仍需完成学生端 `/api/student/certificates*` 的身份隔离、列表和详情路由；当前只存在兼容下载路径，属于后续 P0 收尾项。
