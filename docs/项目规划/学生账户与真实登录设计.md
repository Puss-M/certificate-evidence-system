# 学生账户与真实登录设计

## 1. 目标与范围

将现有“输入 `student_no` 查询”的临时学生演示页升级为真实学生账户体系。学生登录后只能查看、下载和分享本人证书；身份判定由 JWT 和数据库绑定完成，不再相信 URL、查询参数或浏览器本地保存的学号。

本次保留并恢复已有学生端业务能力：

1. 我的证书列表和证书详情。
2. 本人证书 PDF 下载和二维码展示。
3. 公共验真链接复制与跳转。
4. 有效、撤销、补发和存证状态展示。
5. 首次登录修改密码、主动退出和管理员重置密码。

不做学生自助注册、邮箱/短信验证、第三方登录、找回密码或公开旧 mock 下载接口。

## 2. 决策记录

| 决策 | 备选 | 原因 |
| --- | --- | --- |
| 在 `users` 表绑定 `student_id` | 独立学生账号表 | 复用现有 JWT、会话撤销、禁用和密码哈希逻辑，改动最小 |
| 管理员批量开通账号 | 学生按学号自助注册 | 避免学号被抢先绑定，且能直接衔接 Excel 导入的学生记录 |
| 用户名使用学号 | 新增独立用户名 | 学号已唯一，教师和管理员可据导入结果核对并分发初始凭据 |
| 初始密码只在开通/重置结果展示一次 | 明文长期保存 | 数据库只存密码哈希，管理员按需重新生成密码 |
| 学生接口从 JWT 解析身份 | 接受 `student_no` 查询参数 | 消除前端可篡改的身份来源和跨学生下载风险 |
| 不恢复 `/api/certificates/*` 旧接口 | 重新公网放行旧下载接口 | 旧接口是 mock/按编号下载，不校验学生身份；真实学生端使用受 JWT 保护的新接口 |

## 3. 数据与认证

1. `User.role` 增加 `STUDENT`；`users.student_id` 为可空且唯一的外键，管理员、教师、审计员保持为空。
2. `users.must_change_password` 标识初始密码或管理员重置后的强制改密状态。
3. 管理员为已导入、尚未绑定账号的学生批量创建 `STUDENT` 用户；用户名固定为 `student_no`，初始密码由服务端生成。
4. 学生使用现有登录接口获得 JWT。登录响应包含角色和是否必须改密；首次登录只允许改密、查询当前账号和退出。
5. 学生改密完成后访问学生中心。禁用学生账号或重置密码会撤销其有效会话。

## 4. 接口与页面

### 管理端

- 批量开通学生账号：仅管理员/教师可执行，返回本次新建账号和一次性初始密码结果。
- 单个重置学生密码：仅管理员/教师可执行，返回一次性新密码。
- 学生列表显示账号状态，避免为已绑定学生重复开户。

### 学生端

- 新增学生登录入口和首次改密页。
- 学生中心不再显示学号输入框；从登录用户加载本人姓名和证书。
- 学生证书 API 保留现有列表、详情、PDF、二维码能力，但取消 `student_no` 查询参数并强制 `STUDENT` JWT。
- 证书详情继续提供“下载 PDF”“二维码展示”“复制验真链接”“打开公共验真页”。撤销和补发状态继续按真实证书记录展示。

## 5. 迁移、兼容与部署

1. 数据库升级仅新增 `users.student_id`、`users.must_change_password` 和所需唯一索引；既有协作账号和学生记录不删除、不重写。
2. 已导入学生通过管理端批量开通，不要求重新导入 Excel。
3. Caddy 继续仅代理学生页面和 `/api/student/certificates*`；接口的访问控制由 FastAPI JWT 执行。旧 `/api/certificates*` 继续保持公网 `404`。
4. 发布前备份数据库、输出目录和 Caddy 配置；发布失败按现有 release 回滚流程恢复。

## 6. 验收标准

1. 管理员导入或选择已有学生后可批量开通，且不能重复绑定。
2. 学生用学号和初始密码登录，首次必须改密；改密后可重新登录。
3. 学生只能看到自己的证书，携带其他 `student_no`、修改 URL 或猜测证书编号均不能访问他人详情、PDF 或二维码。
4. 学生可下载本人 PDF、打开二维码和公共验真链接；有效、撤销、补发状态准确展示。
5. 重置密码和禁用账号后，旧令牌失效。
6. 后端认证/权限/学生证书测试、前端构建、Caddy 校验和线上浏览器回归通过。
